# Neverland → Architectural Builder — технический план

## 1. Цель архитектуры

Сохранить существующий чанковый субстрат (стриминг, immutable-снапшоты, фоновые воркеры
PandaAsync, версии и отбрасывание устаревших результатов, инвалидация соседей) и заменить
только последнюю милю — превращение данных чанка в треугольники.

```text
Voxel/Cell Data
    ↓
Semantic Build Stage
    ↓
Mesh Compilation
```

Существующий код (Scripts/Neverland/src/Model/): `ChunksStorage` (владение чанками,
очереди билдов), `Chunk` 20×80×20, `ChunkMeshSnapshot` (padding 1), `GameContext`
(scheduler + счётчики фоновых задач), `RegionMesh`/`RegionMeshGenerator` (LOD дальних
регионов), `VoxelMeshGenerator` → переименован в `TerrainMeshGenerator`.

## 2. Разделение генераторов

```cpp
class TerrainMeshGenerator;    // земля/скалы: marching cubes; рукотворные блоки: кубы (этап 1)
class BuildingMeshGenerator;   // стены, углы, проёмы, окна, двери, карнизы (этапы 3-5)
class RoofMeshGenerator;       // скаты, коньки, фронтоны (этап 6)
class DecorationMeshGenerator; // декор (после MVP)
```

Промежуточное представление (GeometryPrimitive / ChunkMeshCompiler из исходного плана)
**вводится не раньше этапа фасадов**: для terrain и WallRun-стен достаточно прямой сборки
MeshData. IR появляется вместе с ExtrudedProfile (окно/карниз), когда возникает реальная
общая структура.

## 3. Terrain: marching cubes (этап 1 — в работе)

Типы вокселей делятся на **природные** (grass, ground, stone, sand, sand_stone) и
**рукотворные** (boards, stone_bricks, tree, birch_log, birch_leaves). Природные рендерятся
гладкой поверхностью, рукотворные — кубами (это «конструкции», их сетка видима намеренно),
tall_grass — крест как раньше.

- Скалярное поле: density в узлах решётки = среднее 8 смежных вокселей (0/1 природность);
  iso = 0.5. Рукотворные блоки участвуют в плотности (земля прилегает к фундаментам без
  щелей), но рисуются собственным кубическим проходом.
- Нормали — градиент density-поля (гладкое освещение склонов).
- UV — псевдо-triplanar: планарная проекция по доминантной оси нормали треугольника,
  тайл — из атласа по типу ближайшего воксела. Швы проекций допустимы на этапе 1.
- Коллизия остаётся воксельной (VoxelCharacterController по сетке). Расхождение
  видимой и физической поверхности на склонах — известный компромисс; вариант на
  будущее: ground snap по семплу MC-поверхности.
- Копание/установка (bresenham3D по вокселям) не меняется: правка вокселя → ремеш → 
  поверхность «вминается» естественно.

LOD: `RegionMeshGenerator` (дальние регионы) пока остаётся кубическим — стык near/far
станет заметнее; перевод LOD на прореженное MC-поле — отдельный шаг после этапа 1.

## 4. Модель данных чанка (этап 2)

Terrain-воксели и архитектурные объекты разделяются:

```cpp
struct ChunkSnapshot {
    ChunkCoord coord;
    uint32_t version;
    TerrainVoxelGrid terrain;
    BuildingCellGrid buildingCells;                       // занятость + ссылки, не геометрия
    std::vector<ArchitectureObjectSnapshot> architecture; // multi-cell объекты
};
```

Архитектурный объект: id, тип (Wall, Corner, Window, Door, Cornice, RoofVolume, ...),
origin, sizeInCells, rotation (90°), материал, ParameterBlock.

Правило владения: объект компилируется чанком, содержащим его origin; соседние чанки
хранят только occupancy. Для бесшовных стен/карнизов генератору передаётся
neighborhood-снапшот (соседние чанки).

## 5. Генерация стен (этап 3)

```text
Cell → WallRun → Openings → Surface Parts → MeshData
```

1. Найти последовательности совместимых стен → WallRun.
2. Собрать проёмы (окна/двери), пересекающие стену.
3. Разделить поверхность по проёмам, создать крупные плоскости вместо граней клеток.
4. Добавить откосы проёмов.

## 6. Окно и карниз (этапы 4-5, здесь появляется IR)

Окно — составной объект (проём, каменная рамка, подоконник, деревянная рама, стекло).
Основная рамка — ExtrudedProfile (2D-профиль по пути). Карниз: собрать совместимые
сегменты → непрерывный path → протянуть профиль → угловые соединения.

Стекло/камень/дерево — раздельные material batches: в Panda это несколько
RuntimeMesh-сущностей на чанк, скриптовый API `mesh_Create/Update` уже позволяет,
движковых правок не требуется.

`StaticMeshPrimitive` (капители, сложный декор) в MVP не нужен. Когда понадобится —
развилка: процедурный декор в скриптах / декор-модули как entity с ImportedSubmesh /
ABI чтения меш-данных (последнее — движковая правка, согласовать отдельно).

## 7. Крыша (этап 6)

Отдельная семантическая система, не наклонные воксели. MVP: прямоугольный footprint,
двускатная крыша, фиксированные углы наклона, фронтоны, свес.

## 8. Инвалидация

Существующий `version` сохраняется; добавляются типы изменений (Terrain / Building /
Roof / Decoration): смена параметра окна перестраивает building-меш чанка, не terrain.
Изменение на границе чанка инвалидирует соседей (механика уже есть).

## 9. Потоки

Без изменений: main thread делает снапшоты, worker строит меш, main принимает результат
при совпадении версии (существующий пайплайн ChunksStorage + PandaAsync).

## 10. Этапы

1. **TerrainMeshGenerator + marching cubes** — в работе. Отдельно после: LOD-регионы на MC.
2. **BuildingCellGrid** — строительный слой, multi-cell ownership, разделение мешей
   terrain/building (отдельные сущности).
3. **WallRun-стены** — крупные плоскости, проёмы, откосы. Первый «не воксельный» фасад.
4. **Параметрическое окно + дверь** — вводится IR (ExtrudedProfile), material batches.
5. **Непрерывный карниз** — профиль по пути, углы.
6. **Двускатная крыша** — RoofVolume.
7. **UX строительства** — параллельный трек (призрак, поворот, палитра, build-камера).
8. **Свет по вокселям** — светомап flood-fill (sunlight сверху + blocklight от источников),
   значение света и AO из density — в вершины меша (fragLight/отдельный канал); после этого
   включается обратно цикл дня и ночи (SunCycle сейчас в режиме «вечный день»: enabled=0 —
   фиксированный полдень, ночные минимумы подняты, чтобы ночь не была чёрной).
9. **Оптимизация** — LOD, кэш профилей, объединение примитивов, collision-геометрия.

## 11. Сохранение мира (готово, июль 2026)

Движок: `ApplicationAPI::getPersistentDataPath()` (ABI v20) — редактор отдаёт
`<проект>/.Panda/SaveData`, рантайм — OS-путь (`Application Support`/`%APPDATA%`/Documents)
с именем игры из RuntimeManifest. Neverland: `WorldSave` — бинарный `neverland_world.sav`:
дельты правленых чанков относительно детерминированной генерации (voxel index u16 + type u8;
эталон регенерируется при capture) + позиция/взгляд/слот игрока. Загрузка дельт — до первых
`ensureChunk`; сейв — автосейв каждые 30 с + при выходе (BaseScript::shutdown). Правленые
чанки не выгружаются стримингом (уже было так), поэтому capture нужен только в момент сейва.

## 12. Критерии готовности

- terrain и здания перестраиваются независимо;
- окно занимает несколько клеток и вырезает корректный проём;
- стены объединяются в крупные поверхности;
- карниз непрерывен через несколько клеток;
- объект на границе чанков не дублируется;
- генерация в worker thread, результат отбрасывается при несовпадении версии.
